# Repository Guidelines

## Project Structure & Module Organization

- `src/index.ts` contains the complete TypeScript OpenCode plugin: configuration loading, status collection, tmux-file handling, and event hooks.
- `scripts/openspec-status.sh` is the bundled Bash helper that renders OpenSpec progress. It is invoked by the plugin at runtime.
- `dist/` is generated by TypeScript and is the published package entry point; do not edit it directly.
- `README.md` documents installation, configuration, tmux integration, and troubleshooting. Keep it aligned with user-facing behavior.

## Build, Test, and Development Commands

- `npm install` installs the Node.js development dependencies. Use Node.js 18 or newer.
- `npm run build` runs `tsc`, emitting ESM JavaScript and declarations to `dist/`.
- `npx tsc --noEmit` type-checks without changing generated output; run it while iterating on `src/index.ts`.
- `bash scripts/openspec-status.sh` exercises the helper in an OpenSpec-enabled project. Pass `--all` to render every change.

There is currently no automated test runner. Validate changes by building, running relevant helper-script cases, and manually checking the tmux status-file behavior when it is affected.

## Coding Style & Naming Conventions

Use TypeScript with 2-space indentation, semicolon-free statements, double-quoted strings, and strict typing. Prefer small, focused functions such as `collectGitDiff` and explicit interfaces for configuration data. Use `camelCase` for values and functions, `PascalCase` for types, and `UPPER_SNAKE_CASE` for module-level constants. Preserve the existing defensive error handling around shell commands and filesystem access; the plugin should fail silently rather than interrupt OpenCode.

Keep the Bash script POSIX-like where practical, quote variable expansions, and retain its `shellcheck`-friendly structure. Avoid changing tmux pane-ID sanitization without checking the README troubleshooting notes.

## Commit & Pull Request Guidelines

Use concise, imperative Conventional Commit-style subjects, as in `fix: keep % in pane id` or `docs: add tmux troubleshooting notes`. Keep each commit scoped to one behavior change.

Pull requests should explain the user-visible effect, note configuration or tmux changes, and include command output or screenshots for statusline rendering changes. Link related issues when available and update `README.md` whenever defaults, options, environment variables, or setup instructions change.
